JNI概述

Java Native Interface简称JNI，即Java本地调用，JNI定义了一种Java层与 Native层通信方式，能使我们在Java层调用Native函数（C/C++函数），同时也能使我们在Native层调用Java方法。对我们开发人员来说丰富成熟的Native语言模块及出色的运行效率能使我们的开发工作更快捷，更有效率。

注册的过程实际上是查找Native函数，并调用dvmUseJNIBridge完成nativeFunc赋值的过程，将静态注册方式执行阶段的工作提前到加载库时完成，因此动态注册不仅在函数名称上方便定义，在执行效率上也是略优于静态注册。

JNIEnv介绍

JNIEnv是一个代表JNI环境变量的结构体，其内部提供了一些JNI系统函数，通过这些函数我们可以回调Java方法，操作jobject对象。不过需要强调的是JNIEnv是跟线程相关的变量，不能在线程间共享，你不能再线程A中使用线程B的JNIEnv。一般我们使用的都是函数以参数传递过来的，那么如果在一个静态函数或在后台线程中需要使用到JNIEnv我们该如何获取 ？ 在动态注册的时候，JNI_OnLoad的参数里有一个JavaVM参数，JavaVM是虚拟机在JNI层的代表，每个进程只允许有一个，我们可以保存JavaVM，通过调用AttachCurrentThread获得当前线程的JNIEnv,在线程退出时调用DetachCurrentThread释放资源.

垃圾回收

在我们的helloJni中传递了一个JniTest对象，现在我想把它保存起来，在helloJniRegister中调用JniTest的nativeCallJava方法：

好，我们调用一下：

很遗憾，程序崩溃了

异常提示我们使用了一个无效的jobject,可我们的jnitest明明赋值了的，为什么会无效呢？ 在Java中创建的对象是由垃圾回收器回收和释放内存的，我们传递的JniTest对象是一个局部变量，在helloJni执行完成后就被回收了，所以在JNI层jnitest指向的就是一个无效的地址。那么如何避免这种情况呢？ 为了解决这一问题，JNI提供了三种类型的引用：

Local Reference:
局部引用，在JNI层使用的非全局引用全为局部引用，它包括函数传递进来的jobject及在函数中创建的jobject，函数执行完毕后，这些jobject都将被回收。不过在循环中创建的jobject需要显示调用DeleteLocalRef()去释放引用，不然有可能造成local reference table overflow。

Global Reference:
全局引用，不主动释放，不会被回收。需显示调用DeleteGlobalRef()释放。创建时也许显示调用NewGlobalRef()。

Weak Global Reference:
弱全局引用，在运行过程中有可能被回收，在使用时应调用IsSameObject(env, jobj,NULL)判断是否已被回收。创建和释放是通过 NewWeakGlobalRef() 和 DeleteWeakGlobalRef()

同一个 jobject 对象的不同引用可能拥有不同的值，比如同一 jobject 对象每次调用 NewGlobalRef() 可能返回不同的值；要检查两个引用是否指向同一个 jobject 对象，必须调用 IsSameObject()，而不要使用 == 去比较；用于描述一个 jobject 对象的 32位值可能在方法多次调用后发生变化，而两个不同 jobject对象却可能在多次方法调用拥有相同的值，所以千万不能将 jobject 对象的值当作 key 使用。
